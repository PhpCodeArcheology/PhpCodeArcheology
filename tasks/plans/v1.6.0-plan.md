# Implementierungsplan v1.6.0 — Report-Verzeichnis-Isolation

Erstellt: 2026-03-24
Analyst: Agent Task 1

---

## 1. Ist-Analyse

### 1.1 Aktuelles Verhalten

Alle Report-Klassen setzen `$outputDir = $config->get('reportDir') . DIRECTORY_SEPARATOR`. Das führt dazu, dass alle Reports ins gleiche Wurzelverzeichnis schreiben:

```
reportDir/
├── history.jsonl          ← HistoryService
├── index.html             ← HtmlReport
├── assets/                ← HtmlReport
├── files/, classes/, …    ← HtmlReport
├── report.json            ← JsonReport
├── report.sarif.json      ← SarifReport
└── ai-summary.md          ← AiSummaryReport
```

`HtmlReport::generate()` und `MarkdownReport::generate()` rufen `clearReportDir()` auf, das **alles außer history.jsonl löscht**. Damit werden bei jedem HTML-Run alle JSON/SARIF/AI-Reports zerstört.

### 1.2 Nutzung von ReportTrait

`ReportTrait` wird **nur** von `HtmlReport` und `MarkdownReport` genutzt (via `use ReportTrait;`). Die drei anderen Klassen (`JsonReport`, `SarifReport`, `AiSummaryReport`) implementieren `ReportInterface` direkt ohne den Trait.

### 1.3 Kritische Pfade für history.jsonl

**HistoryService** (`src/Application/Service/HistoryService.php`):
- Lesen (Zeile 21): `$config->get('reportDir') . DIRECTORY_SEPARATOR . 'history.jsonl'`
- Schreiben (Zeile 142): `$config->get('reportDir') . DIRECTORY_SEPARATOR . 'history.jsonl'`

**HtmlReport::generateIndexPage()** (Zeile 135):
- Lesen: `$this->config->get('reportDir') . DIRECTORY_SEPARATOR . 'history.jsonl'`

Alle drei Stellen zeigen auf `reportDir/history.jsonl` — der Root-Pfad. Diese Stellen dürfen **nicht** geändert werden.

### 1.4 MarkdownReport erbt von HtmlReport

`MarkdownReport` ruft `parent::__construct()` auf und überschreibt danach nur `$this->templateDir`. Der Parent-Konstruktor setzt `$this->outputDir` auf `reportDir/`. MarkdownReport muss nach dem Parent-Aufruf `$this->outputDir` überschreiben.

## 3. Änderungen pro Datei

### 3.1 `src/Report/ReportTrait.php`

**Ziel:** Neue Hilfsmethode `getReportSubDir()` hinzufügen, die den Unterverzeichnis-Namen zurückgibt. Jede Klasse, die den Trait nutzt, definiert ihre eigene Implementierung über ein geschütztes Property.

**Änderungen:**

1. Neues geschütztes Property hinzufügen:
   ```php
   protected string $reportSubDirName = '';
   ```

2. Neue Methode `getReportSubDir()`:
   ```php
   protected function getReportSubDir(): string
   {
       return $this->reportSubDirName;
   }
   ```

3. `clearReportDir()`: **Keine inhaltliche Änderung notwendig.** Die Methode löscht bereits nur `$this->outputDir`. Da `$this->outputDir` neu auf das Unterverzeichnis zeigt (z.B. `reportDir/html/`), ist das Verhalten automatisch korrekt. Die Tiefen-Sicherheitsprüfung (≥ 3 Ebenen) bleibt bestehen — bei typischen Pfaden wie `/home/user/project/tmp/report/html/` greift sie korrekt.

4. `deleteDirContents()`: Der bestehende Skip für `history.json`/`history.jsonl` bleibt als Defense-in-Depth. Da history.jsonl nun im Parent-Dir liegt, wird es sowieso nicht gefunden, aber der Guard schadet nicht.

**Keine Signaturänderung von `renderTemplate()` notwendig** — sie schreibt bereits in `$this->outputDir . $outputFile`, was automatisch korrekt wird.

### 3.3 `src/Report/MarkdownReport.php`

**Konstruktor:** Nach `parent::__construct()` `$this->outputDir` und `$this->reportSubDirName` überschreiben.

**Vorher (Konstruktor-Ende):**
```php
parent::__construct(...);
$this->templateDir = dirname(__DIR__, 2) . '/templates/markdown' . DIRECTORY_SEPARATOR;
$this->twigLoader->setPaths($this->templateDir);
if (is_dir($this->templateDir . 'parts')) {
    $this->twigLoader->addPath($this->templateDir . 'parts', 'Parts');
}
```

**Nachher:**
```php
parent::__construct(...);

// Unterverzeichnis für Markdown überschreiben
$this->reportSubDirName = 'markdown';
$this->outputDir = $config->get('reportDir') . DIRECTORY_SEPARATOR . 'markdown' . DIRECTORY_SEPARATOR;
if (!is_dir($this->outputDir)) {
    mkdir(directory: $this->outputDir, recursive: true);
}

$this->templateDir = dirname(__DIR__, 2) . '/templates/markdown' . DIRECTORY_SEPARATOR;
$this->twigLoader->setPaths($this->templateDir);
if (is_dir($this->templateDir . 'parts')) {
    $this->twigLoader->addPath($this->templateDir . 'parts', 'Parts');
}
```

**Achtung:** Der Parent-Konstruktor erstellt bereits `reportDir/html/` (durch den dortigen mkdir-Aufruf). Das ist unerwünscht — ein leeres `html/`-Verzeichnis wird angelegt, obwohl Markdown gerendert wird. **Lösung:** Im Parent-Konstruktor die `mkdir`-Logik nur ausführen, wenn `$this->outputDir` gesetzt ist, oder den mkdir-Aufruf in `generate()` verschieben (bevorzugt). Alternativ: Der leere `html/`-Ordner wird akzeptiert (kein Funktionsfehler, nur kosmetisch unschön). — Empfehlung: Den `mkdir`-Aufruf aus dem Konstruktor von `HtmlReport` in `generate()` verschieben.

## 5. Risiken und Abhängigkeiten

### Risiko 1: MarkdownReport erstellt ungewollt `html/`-Verzeichnis
**Ursache:** Parent-Konstruktor von HtmlReport erstellt `$this->outputDir` via mkdir. Wenn MarkdownReport den Parent aufruft, zeigt outputDir noch auf `html/`.
**Schwere:** Niedrig (nur leerer Ordner, kein Funktionsfehler).
**Lösung:** mkdir-Aufruf aus HtmlReport-Konstruktor in `generate()` verschieben. Dann erstellt nur der aktive Report sein Verzeichnis.

### Risiko 2: Tiefen-Sicherheitsprüfung in clearReportDir()
**Ursache:** `clearReportDir()` verwirft Pfade mit weniger als 3 Verzeichnisebenen.
**Beurteilung:** Unkritisch. `reportDir/html/` hat bei typischen Pfaden wie `/home/user/project/output/html` 5+ Ebenen. Edge Case: Wenn `reportDir` auf `/a/b` gesetzt ist, schlägt die Prüfung auch für `/a/b/html` fehl (3 Slashes = OK). Tatsächlich: `substr_count('/a/b/html', '/') = 3` — die Prüfung ist `< 3`, also wird `/a/b/html` akzeptiert. Kein Problem.

### Risiko 3: Bestehende history.jsonl bei Nutzern
**Beurteilung:** `history.jsonl` liegt weiterhin in `reportDir/` — keine Migration nötig. HistoryService findet sie am alten Ort.

### Risiko 4: Abwärtskompatibilität für bestehende Nutzer
**Ursache:** Nutzer mit altem Format haben `reportDir/index.html`, `reportDir/report.json` etc. Nach Upgrade liegen neue Reports in Unterverzeichnissen, alte Dateien bleiben im Root.
**Auswirkung:** Alte Report-Dateien werden nicht mehr überschrieben/gelöscht. Sie bleiben als Leichen liegen.
**Lösung (Task 5):** Migrations-Hinweis ausgeben wenn `reportDir/index.html` existiert.

### Risiko 5: Tests
Bestehende Tests in `tests/Unit/Report/` mocken möglicherweise `$config->get('reportDir')`. Nach Änderung könnten Assertions auf Dateipfade fehlschlagen, da nun Unterverzeichnisse erwartet werden.
**Lösung:** Alle bestehenden Report-Tests prüfen und anpassen.
