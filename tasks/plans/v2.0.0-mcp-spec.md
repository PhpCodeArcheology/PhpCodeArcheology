# MCP (Model Context Protocol) — Spezifikations-Referenz

**Recherche-Stand: 2026-03-24 | Protokoll-Version: 2025-11-25 (latest)**

---

## 1. Grundlagen: JSON-RPC 2.0 Message Format

MCP basiert vollständig auf **JSON-RPC 2.0**. Alle Nachrichten MÜSSEN UTF-8 kodiert sein.

### 1.1 Request

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {
    "cursor": "optional-cursor-value"
  }
}
```

Regeln:
- `id` MUSS ein String oder Integer sein (niemals `null`)
- `id` DARF in der gleichen Session nicht wiederverwendet werden
- `params` ist optional

### 1.2 Result Response (Erfolg)

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [...]
  }
}
```

### 1.3 Error Response

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "Unknown tool: invalid_tool_name",
    "data": {
      "supported": ["tool_a", "tool_b"],
      "requested": "invalid_tool_name"
    }
  }
}
```

- `data` ist optional, aber empfohlen für Debug-Infos

### 1.4 Notification (keine Antwort erwartet)

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}
```

- Kein `id`-Feld!
- Empfänger DARF KEINE Antwort senden

## 2. Lifecycle: Initialize-Handshake

### 2.1 Initialize Request (Client → Server)

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-11-25",
    "capabilities": {
      "roots": {
        "listChanged": true
      },
      "sampling": {},
      "elicitation": {}
    },
    "clientInfo": {
      "name": "ExampleClient",
      "version": "1.0.0"
    }
  }
}
```

### 2.2 Initialize Response (Server → Client)

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-11-25",
    "capabilities": {
      "logging": {},
      "prompts": {
        "listChanged": true
      },
      "resources": {
        "subscribe": true,
        "listChanged": true
      },
      "tools": {
        "listChanged": true
      }
    },
    "serverInfo": {
      "name": "ExampleServer",
      "version": "1.0.0"
    },
    "instructions": "Optional: Hinweise für den Client"
  }
}
```

### 2.3 Initialized Notification (Client → Server, danach)

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}
```

Der Client sendet diese Notification nach Erhalt der Initialize Response. Danach beginnt der normale Betrieb.

### 2.4 Initialize Error (Version Mismatch)

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "Unsupported protocol version",
    "data": {
      "supported": ["2024-11-05"],
      "requested": "1.0.0"
    }
  }
}
```

### 2.5 Capability-Übersicht

| Seite   | Capability       | Beschreibung |
|---------|-----------------|--------------|
| Client  | `roots`         | Dateisystem-Roots bereitstellen |
| Client  | `sampling`      | LLM-Completion-Requests empfangen |
| Client  | `elicitation`   | User-Input-Requests empfangen |
| Server  | `tools`         | Tools anbieten (`listChanged`: Notifications) |
| Server  | `resources`     | Ressourcen anbieten (`subscribe`, `listChanged`) |
| Server  | `prompts`       | Prompt-Templates anbieten (`listChanged`) |
| Server  | `logging`       | Strukturierte Log-Messages senden |
| Server  | `completions`   | Argument-Autocomplete unterstützen |

## 3. Tools: tools/list und tools/call

### 3.1 tools/list Request

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list",
  "params": {
    "cursor": "optional-cursor-value"
  }
}
```

### 3.2 tools/list Response

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "tools": [
      {
        "name": "analyze_file",
        "title": "PHP File Analyzer",
        "description": "Analyzes a PHP file for code metrics and quality issues",
        "inputSchema": {
          "type": "object",
          "properties": {
            "file_path": {
              "type": "string",
              "description": "Absolute path to the PHP file"
            },
            "include_metrics": {
              "type": "boolean",
              "description": "Include detailed metrics in the response",
              "default": true
            }
          },
          "required": ["file_path"]
        }
      },
      {
        "name": "get_current_time",
        "description": "Returns the current server time",
        "inputSchema": {
          "type": "object",
          "additionalProperties": false
        }
      }
    ],
    "nextCursor": "next-page-cursor"
  }
}
```

**Tool-Felder:**
- `name`: Eindeutiger Bezeichner, 1-128 Zeichen, Regex: `[A-Za-z0-9_\-.]`
- `title`: Optionaler, menschenlesbarer Anzeigename
- `description`: Erklärung für den LLM, wann das Tool aufzurufen ist
- `inputSchema`: JSON Schema des Tool-Inputs (MUSS ein valides JSON Schema Object sein, niemals `null`)
- `outputSchema`: Optional — JSON Schema für strukturierten Output
- `annotations`: Optional — Metadaten über Tool-Verhalten

### 3.3 tools/call Request

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "analyze_file",
    "arguments": {
      "file_path": "/var/www/project/src/Controller/HomeController.php",
      "include_metrics": true
    }
  }
}
```

### 3.4 tools/call Response (Erfolg)

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "Analysis of HomeController.php:\n- Lines of Code: 142\n- Cyclomatic Complexity: 8\n- Methods: 5"
      }
    ],
    "isError": false
  }
}
```

### 3.5 tools/call Response (Tool-Execution-Error)

Wenn das Tool selbst einen Fehler hat (z.B. Datei nicht gefunden), wird `isError: true` zurückgegeben — aber als `result`, nicht als JSON-RPC `error`:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "Error: File not found: /var/www/project/src/Controller/HomeController.php"
      }
    ],
    "isError": true
  }
}
```

### 3.6 tools/call Protocol Error (Unknown Tool)

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "error": {
    "code": -32602,
    "message": "Unknown tool: invalid_tool_name"
  }
}
```

### 3.7 List Changed Notification

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed"
}
```

## 4. Tool Parameter Schemas (JSON Schema)

### Standardversion

Wenn kein `$schema`-Feld angegeben ist, gilt standardmäßig **JSON Schema 2020-12**. Implementierungen MÜSSEN mindestens 2020-12 unterstützen. Draft-07 wird ebenfalls häufig unterstützt.

### Beispiele

**Tool mit Pflicht- und optionalen Parametern:**
```json
{
  "name": "search_code",
  "description": "Search PHP codebase for a pattern",
  "inputSchema": {
    "type": "object",
    "properties": {
      "pattern": {
        "type": "string",
        "description": "Search pattern (regex supported)"
      },
      "file_extension": {
        "type": "string",
        "enum": ["php", "twig", "yaml", "json"],
        "description": "File extension to filter by",
        "default": "php"
      },
      "max_results": {
        "type": "integer",
        "minimum": 1,
        "maximum": 100,
        "default": 20
      }
    },
    "required": ["pattern"]
  }
}
```

**Tool ohne Parameter:**
```json
{
  "name": "get_project_summary",
  "description": "Returns overview of the analyzed PHP project",
  "inputSchema": {
    "type": "object",
    "additionalProperties": false
  }
}
```

**Tool mit `outputSchema` (strukturierter Output):**
```json
{
  "name": "get_health_score",
  "description": "Returns the overall code health score",
  "inputSchema": {
    "type": "object",
    "additionalProperties": false
  },
  "outputSchema": {
    "type": "object",
    "properties": {
      "score": { "type": "number", "minimum": 0, "maximum": 100 },
      "grade": { "type": "string", "enum": ["A", "B", "C", "D", "F"] },
      "summary": { "type": "string" }
    },
    "required": ["score", "grade", "summary"]
  }
}
```

Response mit `structuredContent` (für `outputSchema`):
```json
{
  "jsonrpc": "2.0",
  "id": 5,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{\"score\": 78.5, \"grade\": \"B\", \"summary\": \"Good code quality with minor issues\"}"
      }
    ],
    "structuredContent": {
      "score": 78.5,
      "grade": "B",
      "summary": "Good code quality with minor issues"
    }
  }
}
```

## 5. Content Types in Tool Responses

Alle Content-Typen unterstützen optionale `annotations`.

### 5.1 Text Content

```json
{
  "type": "text",
  "text": "Analysis result as plain text"
}
```

### 5.2 Image Content

```json
{
  "type": "image",
  "data": "base64-encoded-image-data",
  "mimeType": "image/png",
  "annotations": {
    "audience": ["user"],
    "priority": 0.9
  }
}
```

### 5.3 Audio Content

```json
{
  "type": "audio",
  "data": "base64-encoded-audio-data",
  "mimeType": "audio/wav"
}
```

### 5.4 Resource Link (Verweis auf externe Ressource)

```json
{
  "type": "resource_link",
  "uri": "file:///project/src/main.php",
  "name": "main.php",
  "description": "Primary application entry point",
  "mimeType": "text/x-php"
}
```

### 5.5 Embedded Resource (Ressource eingebettet)

```json
{
  "type": "resource",
  "resource": {
    "uri": "file:///project/src/main.php",
    "mimeType": "text/x-php",
    "text": "<?php\n\necho 'Hello World';",
    "annotations": {
      "audience": ["user", "assistant"],
      "priority": 0.7,
      "lastModified": "2025-05-03T14:30:00Z"
    }
  }
}
```

### Annotations-Format

```json
{
  "annotations": {
    "audience": ["user", "assistant"],
    "priority": 0.8,
    "lastModified": "2025-01-12T15:00:58Z"
  }
}
```

- `audience`: `["user"]`, `["assistant"]`, oder `["user", "assistant"]`
- `priority`: 0.0 (unwichtig) bis 1.0 (essenziell)
- `lastModified`: ISO 8601 Timestamp

## 6. Error Codes

### Standard JSON-RPC 2.0 Codes

| Code    | Name              | Bedeutung |
|---------|-------------------|-----------|
| -32700  | Parse Error       | Ungültiges JSON empfangen |
| -32600  | Invalid Request   | JSON-RPC-Struktur ungültig |
| -32601  | Method Not Found  | Methode existiert nicht |
| -32602  | Invalid Params    | Falsche Parameter (auch: unbekanntes Tool) |
| -32603  | Internal Error    | Interner Server-Fehler |

### MCP-spezifische Codes

| Code    | Bedeutung |
|---------|-----------|
| -32002  | Resource not found |

### Beispiel: Fehler mit `data`-Feld

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "Unsupported protocol version",
    "data": {
      "supported": ["2024-11-05", "2025-11-25"],
      "requested": "1.0.0"
    }
  }
}
```

### Zwei Fehler-Typen bei Tools

**1. Protocol Error** (JSON-RPC `error`-Objekt) — bei strukturellen Problemen:
```json
{ "jsonrpc": "2.0", "id": 3, "error": { "code": -32602, "message": "Unknown tool: foo" } }
```

**2. Tool Execution Error** (im `result` mit `isError: true`) — bei fachlichen Fehlern:
```json
{ "jsonrpc": "2.0", "id": 3, "result": { "content": [{"type":"text","text":"API timeout"}], "isError": true } }
```


```json
{
  "jsonrpc": "2.0",

```

## 8. Transports

### 8.1 stdio (Standard — empfohlen für lokale Server)

- Client startet den Server als Subprocess
- Nachrichten gehen über stdin/stdout
- **Jede Nachricht ist eine Zeile** (Newline-delimited, keine eingebetteten Newlines in der JSON-Message)
- stderr darf für Logging verwendet werden
- Stdout DARF NUR valide MCP-Nachrichten enthalten

```
CLIENT stdin  → {"jsonrpc":"2.0","id":1,"method":"initialize",...}\n
SERVER stdout ← {"jsonrpc":"2.0","id":1,"result":{...}}\n
SERVER stderr → [DEBUG] Server started (optionales Logging)
```

### 8.2 Streamable HTTP (für Remote-Server)

- Ein einzelner HTTP-Endpoint (z.B. `https://example.com/mcp`) für POST und GET
- Client sendet JSON-RPC per HTTP POST
- Server kann mit `application/json` (direkt) oder `text/event-stream` (SSE-Stream) antworten
- HTTP-Header: `MCP-Protocol-Version: 2025-11-25` muss bei allen Requests gesetzt sein
- Session-ID: `MCP-Session-Id` im Response-Header (wenn der Server Sessions verwaltet)

## 9. PHP MCP SDK

### Status: Offiziell — Tier 3

Es gibt ein **offizielles PHP SDK** im `modelcontextprotocol`-GitHub-Org.

| Eigenschaft | Details |
|-------------|---------|
| **Packagist** | `mcp/sdk` |
| **GitHub** | `github.com/modelcontextprotocol/php-sdk` |
| **Version** | v0.4.0 (Stand Feb 2026, pre-1.0) |
| **Status** | Experimental, Tier 3 (von 3 Tiers) |
| **Entwickelt von** | PHP Foundation + Symfony-Projekt |
| **Installation** | `composer require mcp/sdk` |

### Minimal-Server-Beispiel

```php
<?php
require_once __DIR__ . '/vendor/autoload.php';

use Mcp\Server;
use Mcp\Server\Transport\StdioTransport;

$server = Server::builder()
    ->setServerInfo('PhpCodeArcheology MCP', '1.0.0')
    ->setDiscovery(__DIR__, ['.'])
    ->build();

$transport = new StdioTransport();
$server->run($transport);
```

### Tool-Registrierung via PHP-Attribute (automatisch)

```php
use Mcp\Attribute\McpTool;

class CodeAnalysisTools
{
    #[McpTool]
    public function health_score(string $project_path): array
    {
        // ...
        return ['score' => 78.5, 'grade' => 'B'];
    }

    #[McpTool]
    public function search_code(string $pattern, string $extension = 'php'): array
    {
        // ...
    }
}
```

### Tool-Registrierung manuell (programmatisch)

```php
$server = Server::builder()
    ->addTool([CodeAnalysisTools::class, 'health_score'], 'health_score')
    ->addTool([CodeAnalysisTools::class, 'search_code'], 'search_code')
    ->build();
```

### Pro / Contra Analyse

**Pro SDK (`mcp/sdk`):**
- Offiziell — wird von Anthropic/PHP Foundation gepflegt
- Abstrahiert das gesamte JSON-RPC-Protokoll
- PHP-Attribute machen Tool-Definition sehr sauber
- STDIO + HTTP Transports vorhanden
- Discovery-Mechanismus + PSR-16 Caching

**Contra SDK / Argumente für manuelle Implementierung:**
- Tier 3 = experimentell, pre-1.0, breaking changes möglich
- Weniger Downloads / Community als Python/TS SDK
- Für einfache stdio-Server ist das Protokoll nicht komplex
- Manuelle Implementierung gibt volle Kontrolle über Message-Format

**Empfehlung:** Das **offizielle SDK `mcp/sdk` ist der bessere Ausgangspunkt**. Die manuelle Implementierung des JSON-RPC-Protokolls (Parse, Dispatch, Encode) ist fehleranfällig und bringt wenig Mehrwert. Das SDK abstrahiert genau das, was wir nicht selbst schreiben wollen. Version 0.4.x ist gut genug für produktiven Einsatz — bei Breaking Changes wird der Upgrade-Aufwand überschaubar sein.

## 10. Claude Code MCP Server Konfiguration

### Methode 1: CLI (empfohlen)

```bash
# stdio-Server (lokaler PHP-Prozess)
claude mcp add --transport stdio php-code-arch -- php /path/to/mcp-server.php

# Mit Environment-Variable
claude mcp add --transport stdio --env PROJECT_ROOT=/var/www php-code-arch -- php /path/to/mcp-server.php

# HTTP-Server (remote)
claude mcp add --transport http my-api https://api.example.com/mcp

# Scope festlegen
claude mcp add --scope project --transport stdio php-code-arch -- php /path/to/mcp-server.php
```

### Methode 2: `.mcp.json` im Projektroot (für Team-Sharing, in git)

```json
{
  "mcpServers": {
    "php-code-arch": {
      "command": "php",
      "args": ["/path/to/mcp-server.php"],
      "env": {
        "PROJECT_ROOT": "${PROJECT_ROOT:-/var/www/project}"
      }
    }
  }
}
```

### Methode 3: `~/.claude.json` (user-scope, alle Projekte)

Gleiche Struktur wie `.mcp.json`, aber global im Home-Verzeichnis.

### HTTP-Server in `.mcp.json`

```json
{
  "mcpServers": {
    "remote-api": {
      "type": "http",
      "url": "${API_BASE_URL:-https://api.example.com}/mcp",
      "headers": {
        "Authorization": "Bearer ${API_KEY}"
      }
    }
  }
}
```

### Scope-Übersicht

| Scope     | Speicherort              | Gültig für |
|-----------|--------------------------|------------|
| `local`   | `~/.claude.json`         | Aktuelles Projekt, nur du |
| `project` | `.mcp.json` im Projektroot | Alle Team-Mitglieder |
| `user`    | `~/.claude.json`         | Alle deine Projekte |

### Management-Befehle

```bash
claude mcp list          # Alle konfigurierten Server
claude mcp get my-server # Details zu einem Server
claude mcp remove my-server
/mcp                     # Status-Check innerhalb Claude Code
```

### Nützliche Umgebungsvariablen

```bash
MCP_TIMEOUT=10000            # Startup-Timeout in ms
MAX_MCP_OUTPUT_TOKENS=50000  # Output-Limit erhöhen (default: ~10.000 Tokens)
```

## 11. Vollständige Beispiel-Session (stdio)

```
# 1. Client sendet Initialize
→ {"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-11-25","capabilities":{"roots":{"listChanged":true}},"clientInfo":{"name":"claude-code","version":"1.0"}}}

# 2. Server antwortet
← {"jsonrpc":"2.0","id":1,"result":{"protocolVersion":"2025-11-25","capabilities":{"tools":{"listChanged":false}},"serverInfo":{"name":"php-code-arch","version":"1.0.0"}}}

# 3. Client sendet Initialized Notification
→ {"jsonrpc":"2.0","method":"notifications/initialized"}

# 4. Client fragt Tools ab
→ {"jsonrpc":"2.0","id":2,"method":"tools/list"}

# 5. Server listet Tools
← {"jsonrpc":"2.0","id":2,"result":{"tools":[{"name":"health_score","description":"Returns overall code health score","inputSchema":{"type":"object","additionalProperties":false}},{"name":"search_code","description":"Search in codebase","inputSchema":{"type":"object","properties":{"pattern":{"type":"string"},"extension":{"type":"string","default":"php"}},"required":["pattern"]}}]}}

# 6. Client ruft Tool auf
→ {"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"health_score","arguments":{}}}

# 7. Server antwortet
← {"jsonrpc":"2.0","id":3,"result":{"content":[{"type":"text","text":"Health Score: 78.5/100 (Grade B)"}],"isError":false}}
```
