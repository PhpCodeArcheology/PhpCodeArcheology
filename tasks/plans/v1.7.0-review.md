## 1. Kritische Bugs

### 🔴 BUG 1: `gatherData()` wird zweimal aufgerufen — 50% aller Nodes/Edges sind Duplikate

**Nachweis aus Smoke-Test:**
```
Total edges: 1436
Duplicate edges: 718  ← genau 50%!
```
Jede Edge-Kombination (source, target, type) erscheint exakt zweimal. Betroffen: alle Edge-Typen.

**Ursache:**

`ReportDataProviderTrait::__construct()` (Zeile 24) ruft `$this->gatherData()` bereits im Konstruktor auf:
```php
// ReportDataProviderTrait.php:24
$this->gatherData();  // ← erste Ausführung bei new GraphDataProvider(...)
```

`GraphReport::generate()` ruft `gatherData()` zusätzlich explizit auf:
```php
// GraphReport.php:39
$graphDataProvider = $this->dataProviderFactory->getGraphDataProvider();
$graphDataProvider->gatherData();  // ← zweite Ausführung: BUG!
$graphData = $graphDataProvider->getGraphData();
```

Da `$this->nodes`, `$this->edges`, etc. Arrays sind, werden beim zweiten Aufruf alle Einträge nochmals angehängt.

**Alle anderen Reports** (JsonReport, HtmlReport etc.) rufen nur `getTemplateData()` auf — nie explizit `gatherData()`. Das Pattern ist klar: den expliziten `gatherData()`-Call in `GraphReport::generate()` entfernen.

**Fix:**
```php
// GraphReport.php - gatherData() entfernen:
$graphDataProvider = $this->dataProviderFactory->getGraphDataProvider();
// $graphDataProvider->gatherData();  ← diese Zeile löschen
$graphData = $graphDataProvider->getGraphData();
```

**Warum fängt der Test das nicht?**
`GraphReportTest` mockt `gatherData()` mit `shouldReceive('gatherData')` (Rückgabe null). Der zweite Aufruf wird vom Mock still geschluckt. Der Test ist blind für diesen Bug.

### 🔴 BUG 2: Cluster `nodeIds` enthalten falsche IDs — Cluster sind komplett wertlos

**Nachweis aus Smoke-Test:**
```json
{
  "id": "package:PhpCodeArch\\Metrics",
  "nodeIds": ["class:0", "class:1", "class:2", ...]
}
```

Class-Nodes im Graph haben IDs wie `class:xbc91c5df` (Hash-basiert). Cluster-nodeIds enthalten dagegen `class:0`, `class:1` usw. Diese IDs existieren nicht als Nodes → Cluster verweisen auf nicht-existente Nodes.

**Ursache** (`GraphDataProvider.php`, Zeile 101–106):
```php
$classesInPackage = $packageCollection->getCollection('classes');
if ($classesInPackage !== null) {
    foreach ($classesInPackage->getAsArray() as $classId => $className) {
        $clusterNodeIds[] = 'class:' . $classId;  // $classId ist numerischer Index, nicht IdentifierString!
    }
}
```

`PackageMetricsCollection->getCollection('classes')->getAsArray()` liefert `[0 => 'ClassName', 1 => 'OtherClass', ...]` — also numerisch indizierte Werte. Der Code verwendet den **Array-Index als ID**, nicht den Identifier-String.

**Abhilfe:** Entweder die `$className` in `$nameToId` nachschlagen, oder sicherstellen, dass die 'classes'-Collection auf PackageMetricsCollection mit Identifier-Strings als Keys befüllt wird (wie beim ProjectCollection-'classes'-Collection).

```php
// Korrekte Variante:
foreach ($classesInPackage->getAsArray() as $classId => $className) {
    $resolvedId = $nameToId[$className] ?? null;
    if ($resolvedId !== null) {
        $clusterNodeIds[] = 'class:' . $resolvedId;
    }
}
```

**Warum fängt der Test das nicht?**
`GraphDataProviderTest` setzt die 'classes'-Collection manuell mit korrekten Keys:
```php
$pkg->setCollection('classes', new ClassNameCollection([$classId => 'FooClass']));
```
Dieser Test nutzt `$classId` (echten Identifier-String) als Key — genau das, was die Produktions-Implementation NICHT tut. Der Test testet eine andere Datensituation als die Realität.

## 2. Hohe Priorität

### 🟠 FALSCHE METRIC KEY: `paramCount` statt `parameterCount` — alle `params` sind null

**Nachweis:** 1346 Function-Nodes, `params`-Wert null bei ALLEN.

**Ursache:** `ParameterAnalyzer::handleParameters()` speichert:
```php
'parameterCount' => $totalParams,  // ← tatsächlicher Key
```

`GraphDataProvider::processFunctionCollection()` liest:
```php
'params' => $collection->get('paramCount')?->getValue(),  // ← falscher Key
```

**Fix:**
```php
'params' => $collection->get('parameterCount')?->getValue(),
```

---

### 🟠 `mi` ist null für alle 336 Klassen-Nodes

**Nachweis:** `null_mi = 336/336` — kein einziger Klassen-Node hat einen `mi`-Wert.

**Ursache:** Der MaintainabilityIndex wird wahrscheinlich nur auf `FileMetricsCollection` berechnet, nicht auf `ClassMetricsCollection`. Der Analyse-Plan hat das fälschlicherweise als ✅ markiert.

**Empfehlung:** Den `mi`-Eintrag aus den Klassen-Node-Metriken entfernen (oder als null dokumentieren), bis der Calculator erweitert wird. Im aktuellen Zustand ist das Feld nur Rauschen.

## 3. Mittlere Probleme

### 🟡 Method-Nodes erscheinen als Function-Nodes ohne Kontext

`FunctionMetricsCollection` wird für BEIDE Typen genutzt (MetricCollectionFactory mappt MethodCollection → FunctionMetricsCollection). Von den 1346 "function" Nodes im Graph sind ~1300 eigentlich Methoden.

**Problem:** Der Consumer des Graphs kann Methoden nicht von Funktionen unterscheiden, und kann keine Klassen-Methoden-Beziehung auflösen. Es gibt kein `functionType`-Feld im Output.

**Daten wären vorhanden:** `collection->get('functionType')?->getValue()` (aus IdentifyVisitor — setzt `'function'`), und `collection->getPath()` liefert bei MethodCollection den Klassen-Namen statt des Dateipfads.

**Empfehlung:** `functionType` und ggf. `parentClass` (aus `getPath()` bei Methoden) zum function-Node hinzufügen, oder Methoden-Nodes ganz weglassen (sie wurden bisher nicht im Graph-Schema vorgesehen).

---

### 🟡 Tests maskieren Bug 1 durch fehlerhafte Mock-Erwartung

In `GraphReportTest.php`:
```php
$graphDataProvider->shouldReceive('gatherData');  // ← absorbiert den zweiten Aufruf still
```

Korrekte Erwartung sollte sein:
```php
$graphDataProvider->shouldReceive('gatherData')->once();  // ← würde Bug 1 aufdecken!
```

`file`-Nodes und `contains`-Edges wurden bewusst entfernt. Die Test-Fixture sollte aktualisiert werden, auch wenn der Test selbst nur prüft dass Nodes/Edges korrekt ins JSON wandern.

### 🔵 Author-IDs mit Leerzeichen: `"author:Marcus Kober"`

Node-IDs mit Leerzeichen können in manchen Graph-Visualizern (Cytoscape, Gephi, d3) problematisch sein. Erwägen: normalisieren zu `author:marcus-kober` oder URL-encodieren.

---

### 🔵 `_global`-Package mit Null-Metriken erscheint als Node

```json
{"id": "package:_global", "metrics": {"abstractness": null, "instability": null, ...}}
```

Der `_global`-Package ist ein internes Konstrukt für Klassen ohne Namespace. Als Package-Node mit null-Metriken ist er für Graph-Consumer verwirrend. Erwägen: filtern falls alle Metriken null.
