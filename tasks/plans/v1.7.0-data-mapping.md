## 1. MetricsController — Zugriffsmuster

### Collection-Typen (`MetricCollectionTypeEnum`)
```
ProjectCollection   → einmalig; Identifier = "ProjectCollection"
FileCollection      → pro Datei; Identifier = FileIdentifier::ofPath($path)
ClassCollection     → pro Klasse; Identifier = FunctionAndClassIdentifier::ofNameAndPath($name, $path)
FunctionCollection  → pro Funktion; Identifier = FunctionAndClassIdentifier::ofNameAndPath($name, $path)
MethodCollection    → pro Methode; Identifier = FunctionAndClassIdentifier::ofNameAndPath($name, $classPath)
PackageCollection   → pro Paket; Identifier = $packageName (plain string)
```

### Wichtige Zugriffsmethoden
```php
// Alle Collections eines Typs laden (z.B. alle Klassen):
$classes = $metricsController->getMetricCollectionsByCollectionKeys(
    MetricCollectionTypeEnum::ProjectCollection, null, 'classes'
);

// Einzelne Collection per Identifier-String:
$collection = $metricsController->getMetricCollectionByIdentifierString($identifierString);

// Untergeordnete Collection (z.B. Abhängigkeiten einer Klasse):
$deps = $metricsController->getCollectionByIdentifierString($identifierString, 'dependencies');

// Metric-Value per Identifier-String:
$value = $metricsController->getMetricValueByIdentifierString($identifierString, 'instability');

// Alle Collections iterieren:
foreach ($metricsController->getAllCollections() as $collection) { ... }
```

### Node-Typ: `author`
**Quelle:** Keines — FEHLT KOMPLETT

| Graph-Metrik | Property-Key | Herkunft | Verfügbar? |
|---|---|---|---|
| `commitCount` | — | GitAnalyzer | ❌ nur per-file aggregierbar |
| `filesChanged` | — | GitAnalyzer | ❌ nur per-file aggregierbar |

**Was vorhanden ist:** Auf jeder `FileMetricsCollection` liegt `gitAuthors` (array of strings).
Autoren-Nodes müssen durch Aggregation über alle Dateien aufgebaut werden.

**Wie ergänzen:**
```php
// Autoren aus allen File-Collections aggregieren:
$authors = [];
foreach ($metricsController->getAllCollections() as $collection) {
    if (!$collection instanceof FileMetricsCollection) continue;
    $fileAuthors = $collection->get('gitAuthors')?->getValue() ?? [];
    foreach ($fileAuthors as $authorName) {
        if (!isset($authors[$authorName])) {
            $authors[$authorName] = ['commitCount' => 0, 'filesChanged' => 0];
        }
        $authors[$authorName]['filesChanged']++;
        $authors[$authorName]['commitCount'] += $collection->get('gitChurnCount')?->getValue() ?? 0;
    }
}
// Kein AuthorMetricsCollection-Typ vorhanden → muss als Plain-Array im GraphDataProvider leben
```

## 4. Lücken und fehlende Daten

### Lücke 1: Name→ID Auflösung (KRITISCH)
**Problem:** Viele Collections (`extends`, `interfaces`, `traits`, `usedClasses`, `dependencyCycleClasses`) speichern Klassen als **Namen-Strings**, nicht als Identifier-Strings. Um Graph-Edges korrekt zu bauen, muss eine Name→ID Map aufgebaut werden.

**Lösung:** Im `GraphDataProvider` beim Start eine Map aufbauen:
```php
$nameToId = [];
foreach (['classes', 'interfaces', 'traits', 'enums'] as $key) {
    $collection = $metricsController->getCollection(
        MetricCollectionTypeEnum::ProjectCollection, null, $key
    );
    foreach ($collection->getAsArray() as $id => $name) {
        $nameToId[$name] = $id;
    }
}
// Nutzung: $targetId = $nameToId[$className] ?? null;
```

---

### Lücke 2: Autor-Nodes (MITTEL)
**Problem:** Kein `AuthorMetricsCollection`-Typ. Autoren-Daten sind nur als String-Arrays pro Datei gespeichert (`gitAuthors`).

**Lösung:** Im `GraphDataProvider` über alle FileCollections iterieren und Autoren-Nodes on-the-fly aggregieren:
```php
$authorNodes = [];
foreach ($fileCollections as $fileCollection) {
    $churnCount = $fileCollection->get('gitChurnCount')?->getValue() ?? 0;
    foreach ($fileCollection->get('gitAuthors')?->getValue() ?? [] as $authorName) {
        $authorNodes[$authorName]['commitCount'] = ($authorNodes[$authorName]['commitCount'] ?? 0) + $churnCount;
        $authorNodes[$authorName]['filesChanged'] = ($authorNodes[$authorName]['filesChanged'] ?? 0) + 1;
    }
}
```
**Einschränkung:** `commitCount` wird als Summe der `gitChurnCount` aller Dateien approximiert — nicht exakt, da mehrere Autoren pro Commit gezählt werden.

---

### Lücke 3: Package-Cohesion (NIEDRIG)
**Problem:** `cohesion` ist in der Ziel-Graph-Struktur vorgesehen, wird aber in keinem Calculator berechnet.

**Lösung:** Als `null` oder `0` im Graph-Node belassen; spätere Erweiterung im PackageCalculator.

---

### Lücke 4: Class→Method Containment (NIEDRIG)
**Problem:** `MethodCollection` existiert, aber die `contains`-Edge `Class→Method` ist schwächer dokumentiert. `ClassMemberAnalyzer` befüllt die Collections, aber ein direktes "welche Methoden hat Klasse X?" ist nicht per Collection auf der ClassMetricsCollection verfügbar.

**Lösung:** Methods iterieren über `getAllCollections()` und `MethodCollection`-Instanzen filtern; `getPath()` liefert den Klassen-Identifier als Pfad.

---

### Lücke 5: Instabilitäts-Edge-Richtung für `depends_on` (NIEDRIG)
**Problem:** `usedClasses` enthält auch externe Klassen (außerhalb des Projekts). Für den Graph sind nur intra-Projekt-Kanten relevant.

**Lösung:** `usesInProject` (Metric-Value aus CouplingCalculator) verwenden statt `usedClasses`.
