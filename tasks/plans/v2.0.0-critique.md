# v2.0.0 MCP Server — Kritisches Architektur-Review

**Stand: 2026-03-24 | Erstellt von: Kritiker-Agent**
**Geprüfte Artefakte:** v2.0.0-mcp-spec.md, v2.0.0-architecture.md, Application.php, DataProviderFactory.php, MetricsController.php, CliOutput.php, ArgumentParser.php, MetricsCollectionInterface.php, ClassMetricsCollection.php, FunctionAndClassIdentifier.php

---

## Gesamtbewertung: 🟡 GELB

Der Architekturplan hat eine solide Grundstruktur und korrekte MCP-Protokoll-Konformität. Allerdings gibt es **drei kritische Implementierungsfehler** die zur Laufzeit sofort brechen würden, sowie mehrere mittlere Risiken. Die gute Nachricht: Alle Probleme sind vor der Implementierung behebbar.

---

## 1. Kritische Schwachstellen (Severity: BLOCKER)

### 🔴 C1: StderrOutput überschreibt nicht-existente Methode

**Plan (Architecture.md, Abschnitt 3.3):**
```php
class StderrOutput extends CliOutput
{
    protected function write(string $text): void
    {
        fwrite(STDERR, $text);
    }
}
```

**Realität (CliOutput.php):**
`write()` existiert **nicht** in CliOutput. Die Klasse hat folgende Output-Methoden:
- `out(string $message)` → `file_put_contents('php://stdout', $message)`
- `outNl(string $message = '')` → ruft `out()` auf
- `cls()` → schreibt ANSI-Escape-Codes auf stdout

StderrOutput müsste mindestens `out()` überschreiben (und `cls()` deaktivieren), nicht `write()`. Mit dem aktuellen Plan würde die Analysis-Output trotzdem auf stdout gehen und den JSON-RPC-Stream korrumpieren.

**Fix:** `out()` überschreiben, `cls()` ebenfalls überschreiben (ANSI-Escapes auf stderr sind harmlos, aber zur Sicherheit):
```php
class StderrOutput extends CliOutput
{
    public function out(string $message): static
    {
        fwrite(STDERR, $message);
        return $this;
    }

    public function cls(): static
    {
        return $this; // ANSI clear-line auf stderr weglassen
    }
}
```

### 🔴 C2: MetricsTool ruft getName()/getPath() auf dem falschen Interface auf

**Plan (Architecture.md, Abschnitt Tool 3 & 9):**
```
Implementierungshinweis: Iteriere alle Collections, matche getName() / getPath() gegen query.
```
und
```
DataProvider: MetricsController direkt (getAllCollections() + Pattern-Matching auf getName() / getPath())
```

**Realität (MetricsCollectionInterface.php):**
```php
interface MetricsCollectionInterface
{
    public function getIdentifier(): IdentifierInterface;
}
```

Das Interface hat **nur** `getIdentifier()`. `getName()` existiert auf konkreten Klassen wie `ClassMetricsCollection`, aber nicht auf der Interface. Außerdem: `FunctionAndClassIdentifier.__toString()` gibt `'x' + crc32(path + name)` zurück — einen Hash, keinen lesbaren Namen.

Das bedeutet: Ein direkter Call `$collection->getName()` kompiliert nicht (PHP würde fatal error werfen wenn die Methode auf dem Interface nicht existiert, statische Analyse würde warnen). Die Implementierung von `get_metrics` und `search_code` muss `instanceof`-Checks verwenden:

```php
foreach ($this->metricsController->getAllCollections() as $collection) {
    if ($collection instanceof ClassMetricsCollection) {
        $name = $collection->getName();
        // path ist nicht öffentlich auf ClassMetricsCollection!
        if (stripos($name, $query) !== false) { ... }
    } elseif ($collection instanceof FileMetricsCollection) {
        // FileMetricsCollection hat getPath()
        ...
    }
}
```

**Das ist lösbar**, aber der Plan muss das explizit dokumentieren und die Interface-Grenzen anerkennen.

### 🔴 C3: Namespace-Fehler in dispatchCommand()

**Plan (Architecture.md, Abschnitt 5.2):**
```php
'mcp' => (new Command\McpCommand($this))->execute($config, $output, $formatter),
```

**Realität (Application.php, bestehender Code):**
```php
namespace PhpCodeArch\Application;
// ...
'init'     => (new Command\InitCommand())->execute(...),
'baseline' => (new Command\BaselineCommand($this))->execute(...),
```

Der relative Namespace `Command\` in `Application.php` löst nach `PhpCodeArch\Application\Command\` auf. Aber `McpCommand` liegt laut Plan in `PhpCodeArch\Mcp\Command\McpCommand`.

Resultat: PHP Fatal Error — `PhpCodeArch\Application\Command\McpCommand` wird nicht gefunden.

**Fix:** Entweder expliziter `use`-Statement:
```php
use PhpCodeArch\Mcp\Command\McpCommand;
// ...
'mcp' => (new McpCommand($this))->execute($config, $output, $formatter),
```
oder vollqualifizierter Name:
```php
'mcp' => (new \PhpCodeArch\Mcp\Command\McpCommand($this))->execute($config, $output, $formatter),
```

## 2. Hohe Schwachstellen (Severity: HIGH — vor Implementierung klären)

### 🟠 H1: SDK `addTool()` API unklar — 2 vs 3 Parameter

**Plan (Architecture.md, Abschnitt 1):**
```php
$server = Server::builder()
    ->addTool([$healthTool, 'getHealthScore'], 'get_health_score', HealthScoreTool::SCHEMA)
    ->build();
```

**MCP-Spec-Doc (v2.0.0-mcp-spec.md, Abschnitt 9):**
```php
$server = Server::builder()
    ->addTool([CodeAnalysisTools::class, 'health_score'], 'health_score')  // 2 Parameter!
    ->build();
```

Das sind **unterschiedliche Signaturen**. Der Plan benutzt 3 Parameter (callable, tool_name, schema), die Spec zeigt 2 (callable, tool_name). Außerdem: Der Spec-Code nutzt Class-String (`CodeAnalysisTools::class`) statt Instance (`$healthTool`). Der Plan-Autor hat einen Fallback-Hinweis in einer Fußnote versteckt, aber das ist ein fundamentales Risiko.

**Analyse der Auswirkung:**
- Wenn SDK `addTool()` nur 2 Parameter akzeptiert: Alle SCHEMA-Konstanten (10 Stück) wären obsolet. Stattdessen müssten `#[McpTool]`-Attribute auf die Methoden.
- Wenn SDK nur class-strings (nicht instances) akzeptiert: Das gesamte DI-Konzept bricht zusammen (Tools können dann nicht mit DataProviderFactory/MetricsController injiziert werden über Constructor).

**Empfehlung:** Vor der Implementierung SDK v0.4.x Quellcode auf GitHub lesen und die exakte `addTool()` Signatur verifizieren. Dies sollte der erste Schritt sein, nicht eine Fußnote.

---

### 🟠 H2: Config-Validation für `mcp` wird übersprungen — silent failure

**Plan (Architecture.md, Abschnitt 5.1):**
```php
$commands = ['init', 'compare', 'baseline', 'mcp'];
```

**Realität (Application.php, createConfig(), Zeile 182-189):**
```php
// Skip validation for subcommands (they don't need files)
if (!$config->get('command')) {
    try {
        $config->validate();
    }
}
```

**Problem:** Wenn 'mcp' als Subcommand erkannt wird, wird `config->validate()` übersprungen. Aber `McpCommand::execute()` ruft sofort `$this->application->runAnalysis($config, $stderrOutput)` auf, das `$config->get('files')` braucht.

Wenn kein Pfad angegeben wird (`phpcodearcheology mcp`), setzt ArgumentParser:
```php
$config->set('files', [getcwd() . '/src']);  // Default: ./src
```

Das ist eigentlich OK (der Default existiert). Aber wenn `src/` nicht existiert, schlägt `createFileList()` still fehl oder liefert 0 Dateien — ohne klare Fehlermeldung.

**Fix:** McpCommand sollte am Anfang explizit prüfen:
```php
$files = $config->get('files');
if (empty($files)) {
    fwrite(STDERR, '[MCP] Error: No source files specified.' . PHP_EOL);
    return 1;
}
```

---

### 🟠 H3: ArgumentParser erkennt 'mcp' nur wenn explizit in $commands

**Realität (ArgumentParser.php, Zeile 22):**
```php
$commands = ['init', 'compare', 'baseline'];
```

Wenn 'mcp' nicht in dieser Liste ist und jemand `phpcodearcheology mcp src/` ausführt, wird 'mcp' als **Dateipfad** interpretiert, nicht als Subcommand. Die Analysis würde versuchen, eine Datei namens 'mcp' zu lesen — stilles, verwirrendes Fehlverhalten.

Der Plan erwähnt das korrekt ("'mcp' zur $commands-Liste hinzufügen"), aber die Implikation ist kritisch genug um sie hier explizit zu nennen: Das ist eine notwendige Änderung, ohne die der Feature gar nicht funktioniert.

## 3. Mittlere Schwachstellen (Severity: MEDIUM)

### 🟡 M1: DataProvider wird bei jedem Tool-Call neu instanziiert

**Plan (Architecture.md, Abschnitt 3.2 + Tool-Klassen-Muster):**
```php
public function getHealthScore(): array
{
    $provider = $this->factory->getProjectDataProvider();  // NEU bei jedem Call
    $data = $provider->getTemplateData();
    // ...
}
```

**DataProviderFactory.getProjectDataProvider():**
```php
public function getProjectDataProvider(): ProjectDataProvider
{
    return new ProjectDataProvider($this->metricsController);  // kein Caching
}
```

Bei jedem Tool-Call wird ein neuer DataProvider instanziiert. Die DataProvider rufen `gatherData()` im Constructor auf (via ReportDataProviderTrait). Wenn `gatherData()` teuer ist (z.B. bei `GraphDataProvider` oder `ClassCouplingDataProvider`), führt das zu Verzögerungen bei jedem MCP-Tool-Call.

**Empfehlung:** DataProviders in McpCommand einmalig erstellen und als Properties den Tool-Klassen übergeben — oder DataProviderFactory als lazy Cache implementieren.

---

### 🟡 M2: `get_graph` Response-Größe kann MCP-Limits überschreiten

Bei einem Projekt mit 500+ Klassen kann der Graph-JSON mehrere MB groß sein. Das Standard-`MAX_MCP_OUTPUT_TOKENS`-Limit von Claude Code (ca. 10.000 Tokens ≈ 40KB) wäre zu klein.

Der Plan erwähnt `MAX_MCP_OUTPUT_TOKENS=50000` als Umgebungsvariable, aber das ist eine Client-seitige Konfiguration die der User setzen muss. Besser wäre eine Summary-Mode-Option für `get_graph`:
```json
{
  "summary_only": {
    "type": "boolean",
    "default": false,
    "description": "Return only graph statistics without full JSON data"
  }
}
```

---

### 🟡 M3: Analysis-Startup-Zeit blockiert MCP-Verbindung

Claude Code hat ein Standard-MCP-Timeout (standardmäßig ~10 Sekunden). Bei großen Projekten kann `runAnalysis()` länger dauern. Die `.mcp.json` setzt `MCP_TIMEOUT=30000` (30s), aber:
1. Das Timeout gilt für den gesamten Startup, nicht nur die Analysis
2. Claude Code könnte die Connection vor dem ersten `initialize`-Response als "fehlgeschlagen" markieren
3. Ein 30-sekündiger Startup ist aus UX-Sicht problematisch

**Mögliche Mitigation:** Lazy-Analysis (erst beim ersten Tool-Call analysieren) oder Pre-computed Snapshot-Datei. Das ist ein Architektur-Trade-off der explizit diskutiert werden sollte.

---

### 🟡 M4: `search_code` hat `query` als Required, aber `get_metrics` erfordert auch `entity`

Korrekt implementiert. Aber: Die Tool-Beschreibung für `search_code` sagt "query matched against class name, namespace, or file path". Das wird nur mit `instanceof`-Checks möglich (siehe C2). Hier ist kein Schema-Problem, nur ein Implementierungsdetail.

---

## 4. Niedrige Schwachstellen (Severity: LOW)

### 🟢 L1: SDK Breaking Changes Risiko ist real aber manageable

Version 0.4.x ist pre-1.0. Das PHP SDK ist "Tier 3" (experimentell). Die Nutzung ist auf einen einzigen Eintrittspunkt beschränkt (`McpCommand.php`), was den Upgrade-Aufwand bei Breaking Changes begrenzt. Das Risiko ist akzeptabel.

### 🟢 L2: Security-Risiko minimal

Read-only-Betrieb, alle Daten in-memory. Kein Filesystem-Zugriff während Tool-Calls. `entity`-Parameter in `get_metrics` und `query` in `search_code` werden nur gegen in-memory Daten gemappt — kein Path-Traversal-Risiko. SQL-Injection nicht anwendbar. Das Risikoprofil ist sehr niedrig.

### 🟢 L3: Version-Konstante ist veraltet

`Application::VERSION = '1.6.0'` — für v2.0.0 muss das aktualisiert werden. Kein Blocker, aber ein Detail.

### 🟢 L4: `MetricsTool` hat 2 Tools in einer Klasse

`MetricsTool` implementiert sowohl `get_metrics` als auch `search_code`. Das ist eine kleine Verletzung des Single-Responsibility-Prinzips, aber pragmatisch und vertretbar (beide teilen dieselbe Suchabstraktionslogik).

## 6. Architektur-Bewertung

### Was gut ist ✅

- **Richtige Entscheidung für SDK**: Manuelles JSON-RPC wäre fehleranfällig und wartungsintensiv. SDK ist die richtige Wahl.
- **StderrOutput-Konzept**: Korrekte Isolation von Analysis-Output und JSON-RPC-Stream — das Konzept stimmt, nur die Implementierung (C1) muss korrigiert werden.
- **Tool-Struktur mit SCHEMA-Konstanten**: Tool-Definitionen nah am Code halten ist sauber und wartbar.
- **try/catch \Throwable in allen Tool-Methoden**: Korrekte `isError: true`-Semantik laut MCP-Spec.
- **runAnalysis() Signature passt**: `public function runAnalysis(Config $config, CliOutput $output): array` existiert und gibt `[$metricsController, $problems]` zurück — exakt wie im Plan.
- **DataProviderFactory Constructor API stimmt**: `new DataProviderFactory($metricsController)` ist korrekt.

### Was over-engineered ist ⚠️

- **8 Tool-Klassen für 9 Tools**: Angemessen. Tools sind klein genug, dass eine Klasse pro Tool sinnvoll ist.
- **StderrOutput als separate Klasse**: Minimal aber notwendig. Kein Over-Engineering.
- **Doppelte Deps (DataProviderFactory + MetricsController)**: Einige Tools brauchen MetricsController direkt (MetricsTool), andere nur DataProviderFactory. Man könnte MetricsController über Factory exposen, aber das würde die Factory API aufblähen. Pragmatisch akzeptabel.
