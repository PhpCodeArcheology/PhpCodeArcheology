# v1.8.0 — Spec-Kritik: Knowledge Graph Visualisierung

**Gesamtbewertung: 🟡 GELB** — Konzept solide, aber mehrere schwerwiegende Probleme verhindern eine direkte Implementierung ohne Korrekturen.

---

## Schwachstellen (priorisiert)

---

### #1 🔴 KRITISCHER BUG: `initKnowledgeGraph` wird nie aufgerufen

**Problem:** Das Template in Sektion (c) ist inkonsistent mit dem empfohlenen Pattern aus Sektion (b).

Sektion (b) zeigt das korrekte Pattern:
```html
<script src="knowledge-graph.js"></script>
<script>initKnowledgeGraph({{ graphJson|raw }});</script>
```

Sektion (c) (der tatsächliche Template-Code) macht aber:
```html
<script>const graphData = {{ graphJson|raw }};</script>
<script src="knowledge-graph.js"></script>
<!-- KEIN initKnowledgeGraph(graphData)-Aufruf! -->
```

Die IIFE setzt `window.initKnowledgeGraph = initKnowledgeGraph` — aber niemand ruft es auf. Der Graph wird **niemals gerendert**.

**Lösung:** Template aus Sektion (c) korrigieren:
```html
<script src="{% if isSubdir %}../{% endif %}assets/js/d3.min.js"></script>
<script src="{% if isSubdir %}../{% endif %}assets/js/knowledge-graph.js"></script>
<script>initKnowledgeGraph({{ graphJson|raw }});</script>
```
`const graphData` entfernen — unnötige globale Variable.

### #2 🔴 SCHWERWIEGEND: Method/Function-Nodes standardmäßig aktiv — Graph-Explosion

**Problem:** `activeNodeTypes = new Set(Object.keys(NODE_COLORS))` aktiviert ALLE Node-Typen beim Start, inklusive `method` und `function`.

Konsequenz: Eine Codebase mit 100 Klassen und je 10 Methoden erzeugt:
- 100 class-Nodes
- 1.000 method-Nodes
- 1.000 `declares`-Edges (class → method)
- Plus alle class-to-class-Edges

Das sind **2.000+ Nodes und Edges beim ersten Seitenaufruf**. D3 forceManyBody wird bei dieser Größe deutlich ruckeln (Ziel: <200 Nodes für flüssige 60fps). Der Graph ist außerdem visuell unlesbar.

**Lösung:**
1. Method- und Function-Nodes standardmäßig **deaktiviert** starten (Filter-Chips initial als `.inactive`)
2. Oder: In PHP vor dem JSON-Export eine `classOnly`-View erzeugen (ohne method/function-Nodes und `declares`-Edges), die standardmäßig geladen wird
3. Nodes-Limit einbauen: Ab 300+ sichtbaren Nodes eine Warnung zeigen und Simulation verlangsamen (`alphaDecay(0.05)` statt Default)

### #3 🔴 SCHWERWIEGEND: Dashboard lädt vollen GraphJSON für 4 Zahlen

**Problem:** `generateIndexPage()` soll `graphJson` inline ins Dashboard-HTML schreiben — den kompletten Graph mit allen Nodes, Edges, Clusters, Cycles.

Bei einer mittelgroßen Codebase (200 Klassen, 2.000 Methoden):
- JSON-Größe: 2–5 MB inline im HTML
- Das Dashboard-HTML wird dadurch für jeden Besucher riesig
- **Nur für**: 4 Zahlen + Top-5-Liste!

**Lösung:** Für das Dashboard-Widget nur die nötigen aggregierten Daten in PHP berechnen — kein graphJson:
```php
$graphProvider = $this->dataProviderFactory->getGraphDataProvider();
$graphData = $graphProvider->getGraphData();
$data['kgStats'] = [
    'nodes'    => count($graphData['nodes']),
    'edges'    => count($graphData['edges']),
    'cycles'   => count($graphData['cycles']),
    'packages' => count(array_filter($graphData['nodes'], fn($n) => $n['type'] === 'package')),
    'topConnected' => ... // Top-5 in PHP berechnen, nur id/name/connections übergeben
];
```
Kein `graphJson` ans Dashboard-Template.

### #4 🟠 MITTEL: Edge-Filter unvollständig — 4 Edge-Typen nicht steuerbar

**Problem:** Die Filter-Chips im Template decken nur 4 Edge-Typen ab:
`depends_on`, `extends`, `implements`, `cycle_member`

`EDGE_COLORS` (und damit die tatsächlich erzeugten Edges) umfasst 8 Typen:
`uses_trait`, `belongs_to`, `authored_by`, `declares` — **haben keine Filter-Chips**.

Konsequenz: `authored_by`-Edges (Author→Class) sind immer sichtbar sobald Author-Nodes aktiv sind, aber nicht abschaltbar. `declares`-Edges sind das größte Volumen-Problem (#2) und können nicht separat gefiltert werden.

**Lösung:** Filter-Chips für die fehlenden Typen ergänzen:
```html
<span class="filter-chip" data-filter="edge-type" data-value="uses_trait" ...>— uses_trait</span>
<span class="filter-chip" data-filter="edge-type" data-value="belongs_to" ...>— belongs_to</span>
<span class="filter-chip" data-filter="edge-type" data-value="authored_by" ...>— authored_by</span>
<span class="filter-chip" data-filter="edge-type" data-value="declares" ...>— declares</span>
```

### #5 🟠 MITTEL: Theme-Wechsel nach Init aktualisiert Graph nicht

**Problem:** `isLight` wird einmal beim `initKnowledgeGraph()`-Aufruf erfasst. Alle SVG-Farben (Nodes, Edges, Arrow-Marker) werden statisch gesetzt. Das bestehende Dark/Light-Toggle würde den Graph optisch nicht updaten — Nodes hätten falsche Farben im neu gewählten Theme.

**Lösung:** Entweder:
1. Bei Theme-Wechsel `initKnowledgeGraph` erneut aufrufen (wenn kein State-Verlust stört)
2. Farben per CSS-Variable steuern statt via D3-Inline-Attribute:
```css
:root { --node-stroke: rgba(255,255,255,0.15); }
[data-theme=light] { --node-stroke: rgba(0,0,0,0.15); }
```
Und in D3: `.attr('stroke', 'var(--node-stroke)')` — CSS übernimmt Theme-Switching automatisch.

### #6 🟠 MITTEL: `buildGraph()` bei jedem Filter-Klick — Positions-Verlust

**Problem:** Jeder Filter-Klick löst `buildGraph()` aus, was `g.selectAll('*').remove()` ausführt und die D3-Simulation komplett neu startet. Alle Node-Positionen, die der Nutzer manuell per Drag gesetzt hat, gehen verloren. Bei großen Graphs: jeder Klick = mehrere Sekunden Neuberechnung.

**Lösung:** Für einfache Typ-Filter Sichtbarkeit per CSS steuern:
```javascript
nodeGroup.style('display', n => activeNodeTypes.has(resolveNodeType(n)) ? null : 'none');
```
Simulation läuft weiter im Hintergrund, Positionen bleiben. `buildGraph()` nur bei echten Daten-Änderungen (z.B. erstem Load oder CC-Slider) neu aufrufen.

### #7 🟠 MITTEL: D3-Vollbundle (~273 KB) statt selektiver Module

**Problem:** `d3.min.js` (komplette D3 v7) wird heruntergeladen. Die Größe beträgt ~273 KB minified, ~88 KB gzipped. Das Report-HTML ist darauf angewiesen, und der Browser muss es komplett parsen/ausführen.

Tatsächlich benötigt werden nur: `d3-selection`, `d3-force`, `d3-zoom`, `d3-drag` — zusammen ~80 KB minified.

**Lösung:** Entweder:
1. CDN-Link mit Custom Bundle nutzen (wenn Online-Dependency akzeptabel)
2. Beim Build ein D3-Subset einbinden — alternativ akzeptieren, da es cached wird (einmaliger Download pro Browser)

*Anmerkung: Für ein Offline-Report-Tool ist das Vollbundle ggf. der pragmatische Kompromiss. Wichtiger ist Issue #2.*

### #8 🟡 NIEDRIG: Legend zeigt Method/Function nicht — Inkonsistenz zu Filter-Chips

**Problem:** Die `graph-legend` am Ende des Templates listet Nodes: Class, Interface, Trait, Package, Author — aber **nicht** Method und Function. Die Filter-Chips zeigen Method und Function aber als wählbare Optionen.

**Lösung:** Legend um Method/Function ergänzen oder (wenn Issue #2 gelöst durch Deaktivierung) aus den Filter-Chips entfernen und in einem Erweitert-Bereich auslagern.

---

### #9 🟡 NIEDRIG: Abstract-Class-Desaturierung nur in Spec beschrieben, nicht implementiert

**Problem:** Sektion (e) beschreibt: *"Abstract-Classes: leicht desaturiert (opacity 0.7 auf dem Fill)"*. Im JS-Code (`nodeGroup.append('circle')`) gibt es aber nur `.attr('fill-opacity', 0.85)` für ALLE Nodes — keine Unterscheidung nach `flags.abstract`.

**Lösung:** In `knowledge-graph.js`:
```javascript
.attr('fill-opacity', n => n.flags?.abstract ? 0.6 : 0.85)
```

---

### #10 🟡 NIEDRIG: Kein Leerzustand definiert

**Problem:** Wenn alle Node-Filter-Chips deaktiviert werden → leeres SVG, keine Rückmeldung. Bei einer Codebase mit nur 5 Klassen und allen Filtern auf niedrigem CC-Threshold → ebenfalls leerer Graph.

**Lösung:** In `buildGraph()` prüfen:
```javascript
if (visibleNodes.length === 0) {
  g.append('text')
    .attr('x', width / 2).attr('y', height / 2)
    .attr('text-anchor', 'middle')
    .attr('fill', 'rgba(255,255,255,0.3)')
    .text('Keine Nodes sichtbar — Filter anpassen');
}
```

### #11 🟡 NIEDRIG: Barrierefreiheit — Method/Function-Farben nicht unterscheidbar

**Problem:** Method (#94a3b8, slate-400) und Function (#64748b, slate-500) sind beide unsaturierte Grautöne. Für Monochromaten oder Nutzern mit reduzierter Farbwahrnehmung nicht unterscheidbar.

Außerdem: Keine Keyboard-Navigation im Graph, keine ARIA-Labels auf Filter-Chips.

**Lösung (minimal):**
- Method: anderes Symbol (z.B. Raute statt Kreis via `d3.symbol().type(d3.symbolDiamond)`)
- Filter-Chips: `role="checkbox"` und `aria-pressed` Attribut
- Keyboard-Navigation für Graph ist optional (hohe Komplexität)

## Offene Fragen

1. **Soll `declares` überhaupt standard-sichtbar sein?** Wenn Method-Nodes optional/versteckt sind, braucht es `declares`-Edges nur on-demand.

2. **Was ist die erwartete Codebasis-Größe?** Kleine Tools (< 50 Klassen) vs. große Frameworks (> 500 Klassen) brauchen unterschiedliche Default-Ansichten.

3. **Theme-Wechsel nach Seitenladung**: Unterstützt das bestehende Theme-Toggle einen Seiten-Reload? Falls ja, ist Issue #5 automatisch gelöst.

4. **GraphDataProvider-Daten bereits komplett korrekt?** Die Spec sagt "bereits vollständig implementiert" — wurde das verifiziert, inkl. method-Nodes und deren Metriken?

5. **isSubdir-Handling für Knowledge Graph**: Die Seite liegt im Report-Root (`knowledge-graph.html`), nicht in einem Unterverzeichnis. `{% if isSubdir %}../{% endif %}` in den Script-Tags ergibt hier immer den Root-Pfad. Ist das korrekt?
