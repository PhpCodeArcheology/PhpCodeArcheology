# v2.0.0 MCP Server — Architekturplan

**Stand: 2026-03-24 | Erstellt von: Architecture Agent**

---

## 1. Architektur-Entscheidung: SDK vs. Manuell

### Entscheidung: SDK nutzen (`mcp/sdk`) — mit manueller Tool-Registrierung

**Begründung:**

Die manuelle JSON-RPC-Implementierung würde ~400–600 Zeilen fehleranfälligen Low-Level-Code erfordern:
- Message-Framing (Newline-delimited stdio)
- Initialize-Handshake mit Version-Negotiation
- JSON-RPC Parse/Dispatch/Encode Loop
- Ping/Pong, Cancellation-Notifications
- Korrekte Error-Code-Semantik (Protocol Error vs. Tool Execution Error)

Das SDK abstrahiert genau diesen Protokoll-Boilerplate. Version 0.4.0 (Feb 2026, PHP Foundation + Symfony) ist für unseren Use-Case ausreichend stabil. Bei Breaking Changes ist der Upgrade-Aufwand begrenzt, da wir das SDK nur an einem Punkt nutzen (`McpCommand`).

**Aber:** Wir verwenden **keine** Attribute-basierte Discovery (`setDiscovery()`), weil:
- Auto-Discovery erfordert statische Klassen ohne Dependency Injection
- Unsere Tools brauchen `DataProviderFactory` + `MetricsController` als Constructor-Deps
- `addTool([$instance, 'method'], 'tool_name')` gibt uns volle Kontrolle über Tool-Definition

**SDK-Integration:**

```php
// composer.json
"require": {
    "mcp/sdk": "^0.4"
}

// McpCommand.php
use Mcp\Server;
use Mcp\Server\Transport\StdioTransport;

$server = Server::builder()
    ->setServerInfo('PhpCodeArcheology MCP', Application::VERSION)
    ->addTool([$healthTool, 'getHealthScore'],    'get_health_score',    $healthSchema)
    ->addTool([$problemsTool, 'getProblems'],     'get_problems',        $problemsSchema)
    // ... 7 weitere Tools
    ->build();

$server->run(new StdioTransport());
```

## 3. Datenfluss

### 3.1 Startup-Sequenz

```
User: bin/phpcodearcheology mcp src/

1. ArgumentParser::parse()
   → 'mcp' in $commands erkannt
   → config->set('command', 'mcp')
   → config->set('files', ['src/'])

2. Application::run()
   → dispatchCommand('mcp', $config, $output, $formatter)
   → new McpCommand($this)->execute(...)

3. McpCommand::execute()
   a) StderrOutput erstellen (stdout bleibt sauber für MCP)
   b) $application->runAnalysis($config, $stderrOutput)
      → FileList → Analyzer → GitAnalyzer → Calculators → Predictors
      → [$metricsController, $problems] zurück
   c) DataProviderFactory($metricsController) erstellen
   d) Tool-Instanzen mit DataProviderFactory/MetricsController erstellen
   e) SDK Server bauen: addTool() für alle 9 Tools
   f) $server->run(new StdioTransport())  ← blockiert hier bis EOF

4. StdioTransport liest stdin zeilenweise
   → JSON-RPC Handshake (SDK handled automatisch)
   → Danach: Tool-Calls dispatchen
```

### 3.2 Tool-Call-Sequenz (nach Startup)

```
stdin:  {"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"get_problems","arguments":{"severity":"error"}}}

SDK:
  → JSON parsen
  → method='tools/call' → Tool-Handler suchen ('get_problems')
  → ProblemsTool::getProblems(severity: 'error') aufrufen

ProblemsTool::getProblems():
  → new ProblemDataProvider($this->metricsController)
     (Constructor ruft gatherData() automatisch auf)
  → $data = $provider->getTemplateData()
  → Daten für Severity 'error' filtern
  → Text-Response formatieren
  → return ['content' => [['type' => 'text', 'text' => '...']], 'isError' => false]

SDK:
  → Response in JSON-RPC einwickeln
  → auf stdout schreiben

stdout: {"jsonrpc":"2.0","id":3,"result":{"content":[{"type":"text","text":"..."}],"isError":false}}
```

### 3.3 StderrOutput

```php
// MCP-Modus: Analysis-Fortschritt → stderr (stdout ist reserviert für JSON-RPC)
class StderrOutput extends CliOutput
{
    protected function write(string $text): void
    {
        fwrite(STDERR, $text);
    }
}
```

## 4. Tool-Definitionen

### Tool 1: `get_health_score`

**Beschreibung:** Returns the overall code health score and summary for the analyzed PHP project. Use this first to get a high-level overview before diving into specific problems or metrics.

**Parameter:** Keine (`inputSchema: {"type": "object", "additionalProperties": false}`)

**DataProvider:** `ProjectDataProvider` → `getTemplateData()['elements']`

**Relevante Metriken aus ProjectCollection:**
- `healthScore` — Gesundheitsscore 0–100
- `healthGrade` — Grade A/B/C/D/F
- `technicalDebt` — In Stunden
- `overallErrorCount`, `overallWarningCount`, `overallInformationCount`
- `OverallClasses`, `OverallLloc`, `OverallAvgCC`

**Response-Beispiel:**
```
Health Score: 72.4/100 (Grade C)
Technical Debt: 8.3h

Problems: 5 errors, 18 warnings, 12 info
Project: 142 classes, 8,420 LLOC, avg CC: 3.8

Use get_problems to see specific issues, or get_refactoring_priorities for actionable improvements.
```

---

### Tool 2: `get_problems`

**Beschreibung:** Returns code quality problems found in the PHP codebase. Filter by severity (error/warning/info) or problem type. Use after get_health_score to investigate specific issues.

**Parameter:**
```json
{
  "type": "object",
  "properties": {
    "severity": {
      "type": "string",
      "enum": ["error", "warning", "info"],
      "description": "Filter by problem severity level"
    },
    "type": {
      "type": "string",
      "description": "Filter by problem type name (e.g. 'GodClass', 'TooComplex', 'DeadCode')"
    },
    "limit": {
      "type": "integer",
      "minimum": 1,
      "maximum": 200,
      "default": 50,
      "description": "Maximum number of problems to return"
    }
  }
}
```

**DataProvider:** `ProblemDataProvider` → `getTemplateData()['classProblems']`, `['fileProblems']`, `['functionProblems']`

**Implementierungshinweis:** ProblemDataProvider liefert Daten gruppiert nach Entity. Jede Entity hat `problems` (Array von `ProblemInterface`). Filtern nach `$problem->getSeverity()` und `$problem->getType()`.

**Response-Beispiel:**
```
Found 5 errors across 4 entities (showing 5/5):

[error] GodClass: App\Service\UserService
  → Class has 47 methods and 1,240 LLOC. Consider splitting by responsibility.

[error] TooComplex: App\Controller\PaymentController::processPayment()
  → Cyclomatic complexity: 28 (threshold: 10). Extract conditions into smaller methods.

[error] DependencyCycle: App\Repository\OrderRepository ↔ App\Service\OrderService
  → Circular dependency detected. Introduce an interface or service layer.
```

### Tool 3: `get_metrics`

**Beschreibung:** Returns all available metrics for a specific class, file, or function. Use when you need detailed measurements for a particular entity. Provide the class name (e.g. 'UserService' or fully qualified 'App\Service\UserService').

**Parameter:**
```json
{
  "type": "object",
  "properties": {
    "entity": {
      "type": "string",
      "description": "Class name, file path, or function name to look up. Partial match supported."
    },
    "type": {
      "type": "string",
      "enum": ["class", "file", "function", "any"],
      "default": "any",
      "description": "Entity type to search in"
    }
  },
  "required": ["entity"]
}
```

**DataProvider:** `MetricsController` direkt (`getAllCollections()` + Pattern-Matching auf `getName()` / `getPath()`)

**Implementierungshinweis:** Suche alle Collections, finde die mit passendem Namen (case-insensitive, partial match). Gib Typ + alle MetricValues aus.

**Response-Beispiel:**


## 5. McpCommand Integration

### 5.1 ArgumentParser: `mcp` registrieren

```php
// src/Application/ArgumentParser.php — Zeile 22
$commands = ['init', 'compare', 'baseline', 'mcp'];  // + 'mcp' hinzufügen
```

`mcp` erwartet kein Subcommand-Argument (Dateipfade werden wie gewohnt als `files` geparsed).

### 5.2 Application::dispatchCommand(): McpCommand eingehängen

```php
// src/Application/Application.php
private function dispatchCommand(string $command, Config $config, CliOutput $output, CliFormatter $formatter): int
{
    return match ($command) {
        'init'     => (new Command\InitCommand())->execute($config, $output, $formatter),
        'compare'  => (new Command\CompareCommand())->execute($config, $output, $formatter),
        'baseline' => (new Command\BaselineCommand($this))->execute($config, $output, $formatter),
        'mcp'      => (new Command\McpCommand($this))->execute($config, $output, $formatter),  // NEU
        default    => throw new ParamException("Unknown command: $command"),
    };
}
```

### 5.3 McpCommand — vollständige Struktur

```php
// src/Mcp/Command/McpCommand.php

namespace PhpCodeArch\Mcp\Command;

use Mcp\Server;
use Mcp\Server\Transport\StdioTransport;
use PhpCodeArch\Application\Application;
use PhpCodeArch\Application\CliFormatter;
use PhpCodeArch\Application\CliOutput;
use PhpCodeArch\Application\Config;
use PhpCodeArch\Mcp\Output\StderrOutput;
use PhpCodeArch\Mcp\Tools\{ClassListTool, DependenciesTool, GraphTool, HealthScoreTool,
                            HotspotsTool, MetricsTool, ProblemsTool, RefactoringTool};
use PhpCodeArch\Report\DataProvider\DataProviderFactory;

class McpCommand
{
    public function __construct(private readonly Application $application) {}

    public function execute(Config $config, CliOutput $output, CliFormatter $formatter): int
    {
        // 1. Analysis output → stderr (stdout für JSON-RPC reserviert)
        $stderrOutput = new StderrOutput();
        $stderrOutput->setFormatter($formatter);

        // 2. Vollständige Analyse ausführen
        fwrite(STDERR, '[MCP] Starting analysis...' . PHP_EOL);
        [$metricsController, $problems] = $this->application->runAnalysis($config, $stderrOutput);
        fwrite(STDERR, '[MCP] Analysis complete. Starting MCP server.' . PHP_EOL);

        // 3. DataProviderFactory initialisieren (setzt MetricTypes auf alle Values)
        $factory = new DataProviderFactory($metricsController);

        // 4. Tool-Instanzen erstellen
        $healthTool      = new HealthScoreTool($factory, $metricsController);
        $problemsTool    = new ProblemsTool($factory, $metricsController);
        $metricsTool     = new MetricsTool($factory, $metricsController);
        $hotspotsTool    = new HotspotsTool($factory, $metricsController);
        $refactoringTool = new RefactoringTool($factory, $metricsController);
        $dependenciesTool = new DependenciesTool($factory, $metricsController);
        $classListTool   = new ClassListTool($factory, $metricsController);
        $graphTool       = new GraphTool($factory, $metricsController);

        // 5. MCP Server aufbauen
        $server = Server::builder()
            ->setServerInfo('PhpCodeArcheology', Application::VERSION)
            ->addTool([$healthTool,       'getHealthScore'],          'get_health_score',          HealthScoreTool::SCHEMA)
            ->addTool([$problemsTool,     'getProblems'],             'get_problems',              ProblemsTool::SCHEMA)
            ->addTool([$metricsTool,      'getMetrics'],              'get_metrics',               MetricsTool::METRICS_SCHEMA)
            ->addTool([$hotspotsTool,     'getHotspots'],             'get_hotspots',              HotspotsTool::SCHEMA)
            ->addTool([$refactoringTool,  'getRefactoringPriorities'],'get_refactoring_priorities',RefactoringTool::SCHEMA)
            ->addTool([$dependenciesTool, 'getDependencies'],         'get_dependencies',          DependenciesTool::SCHEMA)
            ->addTool([$classListTool,    'getClassList'],            'get_class_list',            ClassListTool::SCHEMA)
            ->addTool([$graphTool,        'getGraph'],                'get_graph',                 GraphTool::SCHEMA)
            ->addTool([$metricsTool,      'searchCode'],              'search_code',               MetricsTool::SEARCH_SCHEMA)
            ->build();

        // 6. stdio-Loop starten (blockiert bis Client-Disconnect/EOF)
        $server->run(new StdioTransport());

        return 0;
    }
}
```

### 5.4 Tool-Klassen-Struktur (Muster)

Alle Tool-Klassen folgen diesem Muster:

```php
// src/Mcp/Tools/HealthScoreTool.php

namespace PhpCodeArch\Mcp\Tools;

use PhpCodeArch\Metrics\Controller\MetricsController;
use PhpCodeArch\Report\DataProvider\DataProviderFactory;

class HealthScoreTool
{
    public const SCHEMA = [
        'description' => 'Returns the overall code health score...',
        'inputSchema' => ['type' => 'object', 'additionalProperties' => false],
    ];

    public function __construct(
        private readonly DataProviderFactory $factory,
        private readonly MetricsController $metricsController
    ) {}

    public function getHealthScore(): array
    {
        try {
            $provider = $this->factory->getProjectDataProvider();
            $data = $provider->getTemplateData();
            $elements = $data['elements'];

            $score = $elements['healthScore']?->getValue() ?? 'N/A';
            $grade = $elements['healthGrade']?->getValue() ?? 'N/A';
            $debt  = $elements['technicalDebt']?->getValue() ?? 0;
            // ... weitere Metriken

            $text = "Health Score: {$score}/100 (Grade {$grade})\n" .
                    "Technical Debt: {$debt}h\n" . /* ... */;

            return ['content' => [['type' => 'text', 'text' => $text]], 'isError' => false];

        } catch (\Throwable $e) {
            return [
                'content' => [['type' => 'text', 'text' => 'Error: ' . $e->getMessage()]],
                'isError' => true,
            ];
        }
    }
}
```

**Hinweis zum SDK-API:** Das genaue Signature von `addTool()` muss beim Implement gegen die tatsächliche SDK v0.4.x API geprüft werden. Alternativ falls `addTool()` nur ein Callable akzeptiert: Tool-Descriptions als `#[McpTool(description: '...')]` Attribute auf die Methoden setzen.

## 6. Error-Handling-Strategie

### 6.1 Zwei Fehler-Typen (laut MCP-Spec)

```
Protocol Error (JSON-RPC error-Objekt):
  → Strukturelle Fehler: ungültiges JSON, unbekannte Method, falscher Tool-Name
  → Wird vom SDK automatisch behandelt
  → Wir implementieren diese NICHT manuell

Tool Execution Error (result mit isError: true):
  → Fachliche Fehler: DataProvider Exception, Entity nicht gefunden, invalide Parameter
  → Wir fangen diese in jedem Tool-Handler mit try/catch
```

### 6.2 Error-Handling in Tool-Methoden

Jede Tool-Methode wrapped ihre Logik in:

```php
try {
    // ... DataProvider nutzen, Daten aufbereiten
    return ['content' => [['type' => 'text', 'text' => $result]], 'isError' => false];
} catch (\Throwable $e) {
    return [
        'content' => [['type' => 'text', 'text' => 'Error: ' . $e->getMessage()]],
        'isError' => true,
    ];
}
```

### 6.3 Spezifische Fehler-Cases

| Situation | Handling |
|-----------|----------|
| DataProvider-Exception | `try/catch` → `isError: true` mit Message |
| Entity nicht gefunden (get_metrics) | `isError: true`: "No class found matching 'Xyz'. Use get_class_list to see available classes." |
| Git-Daten nicht vorhanden (get_hotspots) | Leere Liste + Hinweis: "No git data available. Run analysis on a git repository." |
| Ungültiger Severity-Wert | PHP-Typsystem / enum-Matching deckt dies ab; SDK validiert inputSchema |
| metric_filter mit unbekanntem Key | Graceful ignore — nur bekannte Metriken filtern |
| Analysis schlägt fehl (runAnalysis wirft) | Exception propagiert hoch → stderr + exit 1 (vor Server-Start, kein JSON-RPC nötig) |

### 6.4 Startup-Fehler

Fehler **vor** dem Server-Start (z.B. keine Dateien gefunden, Config-Fehler) schreiben nach stderr und beenden mit Exit-Code 1. Dies ist kein MCP-Fehler, da der Client den Prozess noch nicht kontaktiert hat.

## 8. Offene Punkte / Risks

| Punkt | Risiko | Mitigation |
|-------|--------|-----------|
| SDK `addTool()` API kann von Beispiel abweichen | Mittel | Beim Implement SDK-Source lesen; ggf. Attribute-Ansatz als Fallback |
| `DataProviderFactory` ruft `gatherData()` im Constructor | Low | Das ist das bestehende Verhalten (ReportDataProviderTrait) — kein Problem |
| Große Projekte: Analysis dauert lang vor erstem MCP-Request | Low | stderr informiert Claude Code über Fortschritt; MCP_TIMEOUT anpassen |
| `MetricsTool::searchCode` auf sehr großen Projekten langsam | Low | Limit-Parameter + `getAllCollections()` ist O(n) aber fast genug für CLI-Use |
| `get_graph` Response kann sehr groß werden | Mittel | `MAX_MCP_OUTPUT_TOKENS=50000` env var für Claude Code; Kompression optional |
