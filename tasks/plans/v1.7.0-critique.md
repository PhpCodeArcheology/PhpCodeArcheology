## 1. Kritische Fehler (MUSS korrigiert werden)

### ❌ KRITISCH: `usesInProject` enthält ALLE Abhängigkeiten — Duplikat-Edges entstehen

**Plan-Behauptung (Abschnitt 3, `depends_on` Edge):**
> Option B: `usesInProject` (nur Projekt-interne) bevorzugt für den Graph (filtert externe Klassen)

**Was der Code tatsächlich macht** (`CouplingCalculator::handleClass()`, Zeile 177–208):
```php
$dependencyCollection = $this->metricsController->getCollectionByIdentifierString(
    $identifierString,
    'dependencies'  // ← ALLE Deps: extends + implements + traits + direct usage
);

foreach ($dependencyCollection as $dependency) {
    if (in_array($dependency, $checkArray)) {
        $metricValues['usesInProject'][] = $dependency;  // ← includes extends/implements!
    }
}
```

`usesInProject` wird aus der `dependencies`-Collection befüllt, nicht aus `usedClasses`. Die `dependencies`-Collection enthält **alles**: extends, implements, traits, new, static-calls.

**Konsequenz:** Wenn man `usesInProject` für `depends_on`-Edges benutzt und gleichzeitig `extends`/`implements`/`uses_trait` Edges separat baut, entstehen **Duplikat-Edges**. Klassen mit Vererbung würden mehrfach verbunden.

**Korrekte Lösung:** Für `depends_on` muss `usedClasses`-Collection (aus DependencyVisitor) genutzt werden, die explizit NUR `new`, `static::`, `::CLASS` enthält — ohne extends/implements/traits:
```php
$usedClasses = $metricsController->getCollectionByIdentifierString($classId, 'usedClasses');
// usedClasses = nur direkte Nutzung, KEIN extends/implements/trait
```

Alternativ: `usesInProject` nutzen, aber explizit die bereits als `extends`/`implements`/`uses_trait` erstellten Edges deduplizieren.

---

### ❌ KRITISCH: `DependencyCycleCalculator` nutzt `usedClasses`, nicht `dependencies`

**Plan-Beschreibung (Abschnitt 3, `cycle_member` Edge):**
> **Quelle:** DependencyCycleCalculator — via inDependencyCycle bool + dependencyCycleClasses array

Der Plan verschweigt eine wichtige Einschränkung. `DependencyCycleCalculator::calculate()` (Zeile 61–75):
```php
$usedClasses = $this->metricsController->getCollectionByIdentifierString(
    $identifierString,
    'usedClasses'  // ← NUR direkte Nutzung, NICHT extends/implements!
);
```

**Bedeutung:** Zyklen werden **nur** über `new`/`static::`-Aufrufe erkannt. Eine gegenseitige Vererbung (A extends B, B extends A) wird von diesem Calculator **nicht** als Zyklus erkannt. Der Plan gibt keine Hinweise auf diese Einschränkung.

**Empfehlung:** Im Plan explizit dokumentieren: "cycle_member erfasst nur usage-basierte Zyklen, nicht inheritance-basierte."

## 2. Mittlere Fehler (SOLLTE korrigiert werden)

### ⚠️ `enum` und `anonymous` fehlen in der class-Node-Tabelle

**IdentifyVisitor** (Zeile 344–354) setzt explizit:
```php
'enum' => false,          // Enum-Typ-Flag
'anonymous' => str_starts_with($className, 'anonymous@'),
```

Beide Eigenschaften fehlen im Plan komplett. Besonders `anonymous` ist wichtig für Edge Cases, da anonyme Klassen (`new class {}`) einen speziellen Identifier (`anonymous@<path>:<line>`) bekommen und keine stabilen Namen für Edge-Auflösung haben.

**Empfehlung:** Beide in die class-Node-Tabelle aufnehmen; anonyme Klassen beim Graph-Aufbau explizit überspringen oder als Sonderfall behandeln.

---

### ⚠️ MethodCollection nutzt `$className` als `path` — kein Dateipfad

`DependencyVisitor::saveMethodDependencies()` (Zeile 415–425):
```php
$this->metricsController->setCollection(
    MetricCollectionTypeEnum::MethodCollection,
    [
        'path' => $className,  // ← ClassName als "path", nicht Dateipfad!
        'name' => $methodName,
    ],
    ...
);
```

Der Plan beschreibt Class→Method-Containment als "möglich, aber umständlich" und schlägt vor, über `getAllCollections()` zu iterieren und `MethodCollection`-Instanzen zu filtern. Der Hinweis "`getPath()` liefert den Klassen-Identifier als Pfad" ist korrekt, aber die Implikation für das Edge-Building fehlt: `FunctionAndClassIdentifier::ofNameAndPath($methodName, $className)` wird als Identifier genutzt, nicht `$filePath`.

---

### ⚠️ `Config::VALID_REPORT_TYPES` fehlt als Implementierungsschritt

`Config.php` (Zeile 9):
```php
private const VALID_REPORT_TYPES = ['html', 'markdown', 'json', 'sarif', 'ai-summary'];
```

Ein neuer Report-Typ `graph` (oder was auch immer der Typ heißt) muss hier ergänzt werden. Der Plan erwähnt `ReportFactory` und `DataProviderFactory` als Integrationspunkte, vergisst aber `Config`. Ohne diese Ergänzung würde die Config-Validierung den neuen Typ mit einer Exception ablehnen.

---

### ⚠️ `usesInProject` erfordert ebenfalls Name→ID Auflösung

**Plan (Abschnitt 3, depends_on Option B):**
> Option B: `usesInProject` (nur Projekt-interne) — Aufwand: minimal

`usesInProject` speichert Klassen-**Namen** (Strings), keine Identifier-Strings:
```php
$metricValues['usesInProject'][] = $dependency;  // $dependency ist FQN-String
```

Der Plan trägt Option B mit "minimal" Aufwand ein, aber für Edge-Building ist trotzdem Name→ID Auflösung nötig. Option A (`usedClasses` Collection) ist gleich aufwändig. Der "minimal"-Hinweis ist irreführend.

---

### ⚠️ Interface-zu-Interface `extends` wird falsch modelliert

Bei PHP-Interfaces gilt: Interfaces können nur `extends`, nicht `implements`. Für Interface-Nodes ist daher die `interfaces`-Collection (aus `$node->implements`) **immer leer**. Parent-Interfaces stehen in der `extends`-Collection.

Beim Graph-Aufbau würde eine Interface-Node korrekt über `extends`-Collection verbunden, aber der Plan suggeriert implizit, dass `interfaces`-Collection für Interface-Nodes Daten enthält. Das ist falsch.

## 6. Performance-Überlegungen (nicht im Plan)

`getAllCollections()` gibt ein `array` zurück und lädt **alle** Collections auf einmal in den Speicher. Bei einer Codebase mit 1000 Klassen + gleich viele Funktionen, Methoden, Dateien entstehen 4000+ Objekte. Der `GraphDataProvider` wird potenziell **mehrfach** über `getAllCollections()` iterieren (für File-Nodes, Class-Nodes, Author-Aggregation). Das sind mindestens 3–5 vollständige Iterationen.

**Empfehlung:** Im `GraphDataProvider` einmalig iterieren und alle nötigen Daten in einem Pass sammeln, statt mehrere Iterationen über `getAllCollections()`.

## 7. Abwärtskompatibilität

Der Plan schlägt keine Änderungen an bestehenden Collections vor — das ist gut. Es werden nur neue Collections gelesen, keine modifiziert. Das `DependencyVisitor`-Enhancement (Lücke 3 im ursprünglichen Anforderungsdokument) ist im Plan kein Thema mehr, da die Daten bereits vorhanden sind.
