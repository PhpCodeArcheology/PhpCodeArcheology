# v2.1.0 — Inkrementelle Analyse

Du bist der Team Lead eines Agent-Teams namens "codearch-v2.1". Dein Ziel: Eine inkrementelle Analyse implementieren, die nur geänderte Dateien neu analysiert. Voraussetzung für schnellen MCP-Refresh und späteren `--watch` Mode.

## Voraussetzung

**v2.0.0 muss abgeschlossen sein.** Der MCP Server braucht schnellen Refresh bei Dateiänderungen.

## Ausgangslage

### Aktuelle Analyse-Pipeline

```
Application.run()
  → createConfig()
  → createFileList()            ← FileList sammelt alle PHP-Dateien
  → createMetricController()    ← leerer MetricsController
  → runAnalysis()
      → Analyzer.analyze()      ← Iteriert über ALLE Dateien
          → pro Datei: PHP-Parser → AST → Visitors (IdentifyVisitor, DependencyVisitor, ...)
      → CalculatorService.run() ← Calculators laufen über ALLE Collections
          → CouplingCalculator, DependencyCycleCalculator, HealthScoreCalculator, ...
  → Report generieren oder MCP Server starten
```

### Schlüssel-Dateien

| Datei | Rolle | Relevanz |
|-------|-------|----------|
| `src/Application/Application.php` | Orchestriert Analyse: `run()`, `runAnalysis()`, `createFileList()`, `createMetricController()` | Cache-Integration hier einhängen |
| `src/Application/Analyzer.php` | Führt Analyse pro Datei durch: PHP-Parser → AST → Visitors | Selektive Datei-Analyse |
| `src/Application/FileList.php` | Sammelt PHP-Dateien basierend auf Config (include/exclude/extensions) | Liefert die Datei-Liste |
| `src/Application/Config.php` | Konfiguration mit `get()`, `set()`, `has()`, `validate()` | Neues Flag `--no-cache` |
| `src/Application/ArgumentParser.php` | CLI-Arg Parsing | `--no-cache` / `--fresh` Flag |
| `src/Metrics/MetricsController.php` | Zentraler Daten-Hub, hält alle MetricCollections | Muss aus Cache ladbar sein |
| `src/Calculators/CalculatorService.php` | Orchestriert alle Calculators (Coupling, Cycles, Health Score, ...) | Muss nach inkrementeller Analyse komplett laufen |
| `src/Analysis/DependencyVisitor.php` | Sammelt Dependencies zwischen Klassen | Für Dependency-aware Invalidierung |

### Analyse-Dauer (Problem)

Eine volle Analyse kann bei größeren Projekten mehrere Sekunden bis Minuten dauern. Für den MCP Server (v2.0.0) ist das bei jeder Dateiänderung zu langsam. Die inkrementelle Analyse soll nur geänderte Dateien neu parsen.

### Cache-Strategie (Ziel)

```
.phpcodearch-cache.json         ← Im Projekt-Root
{
  "version": "1.0",
  "createdAt": "2026-03-24T12:00:00+00:00",
  "config": {                   ← Config-Hash für Invalidierung bei Config-Änderungen
    "hash": "abc123"
  },
  "files": {
    "src/UserService.php": {
      "mtime": 1711234567,
      "hash": "sha256:...",     ← Content-Hash für Robustheit
      "metrics": { ... }        ← Gespeicherte Metriken dieser Datei
    }
  },
  "dependencies": {             ← Dependency-Graph für Kaskaden-Invalidierung
    "class:App\\UserService": ["class:App\\UserRepository", "class:App\\Logger"]
  }
}
```

### Inkrementeller Ablauf (Ziel)

```
1. Cache laden (.phpcodearch-cache.json)
2. FileList erstellen (alle PHP-Dateien)
3. Pro Datei: filemtime() vergleichen mit Cache
   → Geändert: markieren für Re-Analyse
   → Neu: markieren für Analyse
   → Gelöscht: aus Cache entfernen
4. Dependency-Kaskade: Wenn Klasse A geändert, alle Klassen die von A abhängen auch markieren
5. Nur markierte Dateien durch Analyzer schicken
6. Gecachte Metriken für unveränderte Dateien in MetricsController laden
7. Calculators KOMPLETT laufen lassen (aggregieren über alle Collections)
8. Cache aktualisieren
```

### Phase 1b — Kritische Plan-Prüfung

#### Task 1b: Cache-Plan kritisch prüfen

- subject: "Cache-Plan kritisch prüfen"
- description: "Du bist der Kritiker des Teams. Lies den Cache-Plan in `tasks/plans/v2.1.0-cache-plan.md` und prüfe ihn kritisch. Cache-Korrektheit ist hier BESONDERS wichtig — ein falscher Cache führt zu falschen Analyse-Ergebnissen, und der User bemerkt es möglicherweise nicht. Prüfe: (1) **Cache-Invalidierung:** Ist die Strategie konservativ genug? Was passiert bei: Branch-Wechsel (git checkout), Rebase, Stash-Pop, Vendor-Updates (composer install)? Wird `filemtime()` allein diesen Fällen gerecht oder braucht es zusätzlich Content-Hashing? (2) **Dependency-Kaskade:** Wie tief geht die Kaskade? Wenn A→B→C und A sich ändert, wird auch C invalidiert? Ist das gewollt? Was bei zirkulären Dependencies? Kann die Kaskade exponentiell werden und den ganzen Cache invalidieren? (3) **Serialisierung:** Sind MetricCollections sicher serialisierbar? Enthalten sie Closures, Ressourcen oder andere nicht-serialisierbare Werte? Was passiert wenn sich die MetricCollection-Klasse zwischen Versionen ändert? (4) **Race Conditions:** Wenn zwei parallele Prozesse den Cache gleichzeitig schreiben (z.B. MCP-Server und ein CLI-Run) — korrupter Cache? Braucht es File-Locking? (5) **Cache-Größe:** Wie groß wird `.phpcodearch-cache.json` bei 1000+ Dateien? 10.000+ Dateien? Kann das zum Problem werden (Parsing-Zeit, Speicher)? (6) **Config-Hash-Strategie:** Welche Config-Keys fließen in den Hash ein? Zu wenige → falscher Cache. Zu viele → unnötige Invalidierung. (7) **Gelöschte Dateien:** Werden Metriken gelöschter Dateien sauber entfernt? Bleiben Dependency-Referenzen auf gelöschte Dateien bestehen? Erstelle dein Review als `tasks/plans/v2.1.0-critique.md` mit: (a) Gesamtbewertung (grün/gelb/rot), (b) Schwachstellen — besonders Cache-Korrektheit-Risiken, (c) Verbesserungsvorschläge, (d) Offene Fragen."

### Phase 2 — Implementierung

#### Task 2: CacheManager implementieren

- subject: "CacheManager für File-Change-Detection"
- description: "Erstelle `src/Application/CacheManager.php`. Lies den Plan in `tasks/plans/v2.1.0-cache-plan.md`. Der CacheManager soll: (1) Cache-Datei: `.phpcodearch-cache.json` im Projekt-Root lesen und schreiben, (2) `loadCache(): array` — Cache-Datei laden, validieren (Version, Config-Hash), bei Invalidität leeren Cache zurückgeben, (3) `detectChanges(array $fileList, array $cache): array` — pro Datei `filemtime()` und optional `sha256_file()` vergleichen. Gibt zurück: `['changed' => [...], 'new' => [...], 'deleted' => [...], 'unchanged' => [...]]`, (4) `resolveDependencyCascade(array $changedFiles, array $dependencyGraph): array` — wenn Datei A geändert wurde und Klasse B von Klasse in A abhängt, dann B auch als geändert markieren. Konservativ: im Zweifel mehr invalidieren, (5) `saveCache(array $fileMetrics, array $dependencyGraph): void` — Cache aktualisieren, (6) `invalidateCache(): void` — Cache-Datei löschen (für `--no-cache`), (7) Config-Hash: Wenn sich die Config ändert (z.B. neue excludes), muss der gesamte Cache invalidiert werden. Nutze `md5(serialize($relevantConfigKeys))`. KONTEXT: Die Cache-Datei liegt im Projekt-Root (dort wo `php-codearch-config.yaml` liegt, NICHT im reportDir)."
